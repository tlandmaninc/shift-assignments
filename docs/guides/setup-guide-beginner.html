Set Up Your Shift Scheduling App

A friendly, step-by-step guide to getting your application running in the cloud — no coding required.

$0 / month ~30 minutes

You're about to set up a shift scheduling application that runs entirely in the cloud — for free! Here's what the finished setup looks like:

Website

What your team sees and uses

Hosted on Vercel (free)

Server

Runs the app's logic behind the scenes

Hosted on Render (free)

Database

Where all your data is safely stored

Hosted on Neon (free)

You'll need free accounts on five services. Each takes about 1 minute to sign up for.

1

Get the Code

~ 3 minutes

First, we need to copy the application's code to your own GitHub account. This is called "forking" — it creates your personal copy that you fully control.

  1. Open the original repository in your browser:
    https://github.com/tlandmaninc/shift-assignments
  2. Look for the "Fork" button near the top-right corner of the page and click it.
  3. On the next screen, leave the default settings and click "Create fork".
  4. Wait a few seconds. GitHub will redirect you to your new copy.
What you should see
Your own copy of the repository at an address like:
https://github.com/YOUR-USERNAME/shift-assignments
The page looks the same as the original, but your username is in the URL now.
What is forking? Forking means creating your own copy of the code. Any changes you make won't affect the original, and you can update your copy at any time.
2

Set Up Your Database (Neon)

~ 5 minutes

The database is where your application stores everything — employees, schedules, shift assignments, and more. Think of it as a spreadsheet that lives on the internet.

  1. Go to neon.tech and sign in (or click "Continue with GitHub").
  2. Click "Create Project" (or "New Project" if you already have one).
  3. Fill in the project details:
    • Project name: shift-assignments (or anything you like)
    • Region: Select "Europe (Frankfurt)" — this is the closest data center
  4. Click "Create Project".
  5. You'll see a dashboard with a connection string. Copy it and save it somewhere safe (like a text file on your desktop). You'll need it in the next step.
What you should see
A dashboard page with a long text that starts with postgresql://. This is your connection string — the address of your database.

Here's what the parts of the connection string mean:

postgresql://neondb_owner:AbCdEf123456@ep-cool-name-12345.eu-central-1.aws.neon.tech/neondb?sslmode=require
yellow = type of database blue = your username pink = your password green = server address purple = database name
Important: Treat this connection string like a password. It contains your database login details. Don't share it publicly or post it online.
3

Deploy Your Server (Render)

~ 10 minutes

The server is the "brain" of your app. It handles logins, creates schedules, runs the AI chat, and talks to the database. Let's get it running on Render.

  1. Go to render.com and sign in.
  2. Click "New" in the top-right, then choose "Web Service".
  3. If asked, connect your GitHub account — click "Connect" and authorize Render to see your repositories.
  4. Find "shift-assignments" in the repository list and click "Connect".
  5. Fill in the basic settings:
    • Name: shift-assignments-backend
    • Region: Frankfurt (EU Central)
    • Branch: main
    • Runtime: Docker
    • Instance Type: Free

Now comes the important part — Environment Variables

What are environment variables? Think of them as a settings form for your app. Each one has a name (like a field label) and a value (what you type in). They tell your app how to connect to services, who the admin is, and more.

Scroll down to the "Environment Variables" section. For each entry below, click "Add Environment Variable", type the name in the first box and the value in the second.

Required Settings

SECRET_KEY Click the "Generate" button next to the value field — this creates a secure random password for your app. If there's no button, type any long random string (32+ characters).
DATABASE_URL Paste the Neon connection string you saved in Step 2 (starts with postgresql://).
ENVIRONMENT Type: production
FRONTEND_URL Type: https://placeholder.vercel.app
(We'll update this to the real address in Step 5)
AI_PROVIDER Type: gemini
GEMINI_API_KEY Paste your Gemini API key from Google AI Studio.
ADMIN_EMAILS Type your email address (the one you use to log in). This makes you the administrator.

Optional Settings (for Google Login)

Skip these if you don't need Google login right now — you can always add them later.

GOOGLE_CLIENT_ID From Google Cloud Console → Credentials → OAuth 2.0 Client ID.
GOOGLE_CLIENT_SECRET Same place as above — the Client Secret value.
  1. After adding all the environment variables, scroll down and click "Create Web Service".
  2. A build log will appear. Wait 3–5 minutes for it to finish. You'll see "Your service is live" when it's done.
  3. Copy the URL shown at the top of the page (it looks like https://shift-assignments-backend.onrender.com). You'll need it for the next step.
Verify it works
Click the URL at the top of your Render dashboard and add /health to the end.
For example: https://shift-assignments-backend.onrender.com/health

You should see: {"status":"healthy"}
Good to know: The free tier puts your server to sleep after 15 minutes of no activity. The first visit after a sleep takes about 30–60 seconds to "wake up" — this is completely normal!
4

Deploy Your Website (Vercel)

~ 5 minutes

Now let's get the website up and running. This is the part your team will actually see and use every day.

  1. Go to vercel.com and sign in.
  2. Click "Add New..." at the top, then choose "Project".
  3. Find "shift-assignments" in the list and click "Import".
  4. Under Framework Preset, make sure "Next.js" is selected (it should be auto-detected).
  5. Important step! Click "Root Directory" → click "Edit" → type frontend → click "Continue".
  6. Expand "Environment Variables" and add one entry:
    API_URL Your Render URL from Step 3.
    Example: https://shift-assignments-backend.onrender.com
  7. Click "Deploy".
What you should see
A build will run for 1–2 minutes, then you'll see a "Congratulations!" page with a preview of your site and your URL (something like https://shift-assignments.vercel.app).
Build failing? The most common cause is forgetting to set the Root Directory to frontend in step 5 above. Go to your project's Settings → General → Root Directory and fix it there.
5

Connect Everything Together

~ 3 minutes

Right now, your server still has a placeholder website address. Let's update it so your server knows where your real website lives.

  1. Go back to Render → click on your shift-assignments-backend service.
  2. Click the "Environment" tab on the left.
  3. Find FRONTEND_URL and change its value from https://placeholder.vercel.app to your actual Vercel URL from Step 4.
    For example: https://shift-assignments.vercel.app
  4. Click "Save Changes". Render will automatically redeploy your server (wait 2–3 minutes).
Why is this step important? This tells your server to only accept requests from your website. It's a security measure that prevents other sites from pretending to be your app.

If you're using Google Login (optional)

If you set up GOOGLE_CLIENT_ID and GOOGLE_CLIENT_SECRET in Step 3, you also need to tell Google about your new website address:

  1. Go to the Google Cloud Console → APIs & Services → Credentials.
  2. Click on your OAuth 2.0 Client ID to edit it.
  3. Under "Authorized redirect URIs", click "Add URI" and enter:
    https://YOUR-VERCEL-URL/api/auth/google/callback
    (Replace YOUR-VERCEL-URL with your actual Vercel address)
  4. Click "Save".
6

Test Your App

~ 2 minutes

Let's make sure everything is working. Go through this checklist and click each circle as you verify it:

  • Visit your Vercel URL
    The login page should appear. If you see a loading screen that never finishes, wait 30–60 seconds (the server may be waking up).
  • Log in with Google (if configured)
    You should be redirected to Google, then back to your app's dashboard.
  • Navigate to the Chat page
    Send a test message. The AI should respond within a few seconds.
  • Check the Employees page
    The page should load (it may be empty if no employees have been added yet — that's fine!).
If all checks pass — congratulations! Your shift scheduling app is live and running!

Frequently Asked Questions

Why does my server take so long to respond?

Render's free tier automatically puts your server to sleep after 15 minutes of no one using it. When someone visits next, the server needs 30–60 seconds to "wake up." This is normal and expected on the free plan.

After the first request wakes it up, subsequent requests will be fast. If you need the server always-on, you can upgrade to Render's paid "Starter" plan (~$7/month).

I see a "CORS error" in my browser

This usually means the FRONTEND_URL environment variable on Render doesn't exactly match your Vercel website address.

Double-check that:

  • The URL starts with https:// (not http://)
  • There's no trailing slash at the end
  • The spelling matches exactly

After fixing it on Render, wait for the automatic redeploy to finish (2–3 minutes).

The AI chat says it's "rate limited"

Google Gemini's free tier allows about 15 requests per minute. If you (or others) are using the chat heavily, you might hit this limit temporarily.

Simply wait about 60 seconds and try again. The limit resets every minute.

My data disappeared after Render redeployed!

If you're seeing this, it likely means the DATABASE_URL environment variable wasn't set correctly (or at all). Without it, the app temporarily stores data on Render's file system, which gets wiped on every deploy.

Go to Render → your service → Environment tab → make sure DATABASE_URL is set to your Neon connection string from Step 2. Once this is set, all data will be safely stored in your Neon database.

How do I update the app when a new version is released?

Both Vercel and Render are connected to your GitHub repository. When new code is pushed to the main branch, they automatically detect the change and redeploy.

To get updates from the original project:

  1. Go to your GitHub repository (your fork)
  2. You'll see a banner saying "This branch is X commits behind." Click "Sync fork" → "Update branch"
  3. Vercel and Render will automatically pick up the changes and redeploy
Is my data safe?

Yes! Neon PostgreSQL is SOC2 compliant, meaning it meets strict security standards. Your data is encrypted at rest (stored securely) and in transit (sent securely). Neon also creates automatic backups.

That said, the free tier has some storage limits. For a typical department scheduling app, you'll have plenty of room.

Can I use a custom domain name?

Yes! Both Vercel and Render support custom domains on their free plans.

  • Vercel: Go to your project → Settings → Domains → Add your domain
  • Render: Go to your service → Settings → Custom Domains → Add your domain

You'll need to update DNS records at your domain registrar (the company where you bought the domain). Both Vercel and Render provide step-by-step instructions when you add the domain.

Remember: If you change your Vercel domain, update the FRONTEND_URL on Render to match!
